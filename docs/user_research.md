# User Research - Theme Sixteen

## AGID-Compliant Public Administration Theme

**Document Version:** 1.0  
**Research Period:** Q4 2025 - Q1 2026  
**Owner:** Theme Product Team

---

## Executive Summary

This document presents comprehensive user research findings for Theme Sixteen, the AGID-compliant public administration theme. Research was conducted between October 2025 and February 2026, involving 45 participants across Italian public administrations, digital agencies, and freelance developers.

### Key Findings Summary

1. **Compliance is Non-Negotiable:** 100% of PA respondents cite AGID compliance as primary selection criterion
2. **Developer Experience Gap:** 78% report frustration with existing theme documentation
3. **Migration Barrier:** 65% hesitate to switch themes due to migration complexity
4. **Training Need:** 82% request structured training programs
5. **Community Value:** 71% willing to pay premium for community-supported solutions

## Key Findings

### Finding 1: Compliance Complexity is Primary Pain Point

**Evidence:**
- 94% of respondents cite "understanding compliance requirements" as challenging
- Average time spent on compliance research: 15 hours/project
- 67% have faced compliance-related project delays

**User Quotes:**
> "I spend more time researching AGID requirements than actually coding. The documentation is scattered and sometimes contradictory."
> — Marco, Municipality Developer (8 years experience)

> "We had to redo an entire module after the accessibility audit. Cost us 3 weeks and €15,000."
> — Giulia, Agency Owner

**Implications:**
- Pre-validated components are highly valued
- Compliance documentation is as important as code
- Audit-ready certification would be a differentiator

---

### Finding 2: Documentation Quality Drives Adoption

**Evidence:**
- 78% rate current theme documentation as "poor" or "fair"
- 85% prefer documentation with working code examples
- 62% have abandoned themes due to poor documentation

**Documentation Preferences:**

| Feature | Importance | Satisfaction (Current) |
|---------|------------|------------------------|
| Working code examples | 95% | 45% |
| Video tutorials | 82% | 30% |
| Search functionality | 88% | 55% |
| Copy-paste snippets | 91% | 60% |
| Troubleshooting guides | 86% | 35% |
| Migration guides | 79% | 25% |

**Implications:**
- Investment in documentation yields direct adoption benefits
- Video content is underserved opportunity
- Migration guides reduce switching barriers

---

### Finding 3: Accessibility Expertise Gap

**Evidence:**
- 71% rate their accessibility knowledge as "basic" or below
- 89% want more accessibility training
- 56% rely on automated tools alone for accessibility testing

**Knowledge Self-Assessment:**

```
Accessibility Knowledge Level (n=150)

Intermediate  ████████░░  24%
Basic         ██████████░░ 35%
Beginner      ████████████  12%
```

**Training Preferences:**
- Hands-on workshops: 67%
- Video courses: 58%
- Written guides: 45%
- Certification programs: 52%
- Office hours/Q&A: 38%

**Implications:**
- Training is a product feature, not just marketing
- Certification program has strong demand
- Accessibility should be "baked in" not "bolted on"

---

### Finding 4: Migration Fear is Real Barrier

**Evidence:**
- 65% interested in Theme Sixteen but concerned about migration
- Average migration time estimate: 40 hours (actual: 15-20 hours)
- 43% have had negative migration experiences

**Migration Concerns:**

| Concern | % Respondents |
|---------|---------------|
| Breaking existing functionality | 78% |
| Time/cost overrun | 72% |
| Team learning curve | 65% |
| Data loss risk | 45% |
| Downtime during migration | 38% |

**Implications:**
- Migration tools and guides are critical
- Migration service offering has demand
- Risk mitigation messaging important

---

### Finding 5: Community Support Highly Valued

**Evidence:**
- 71% willing to pay more for community-supported solutions
- 84% have sought help from online communities
- 62% have contributed back to open source projects

**Support Channel Preferences:**

| Channel | Usage | Satisfaction |
|---------|-------|--------------|
| GitHub Issues | 78% | 72% |
| Discord/Slack | 65% | 81% |
| Stack Overflow | 54% | 68% |
| Email Support | 45% | 75% |
| Documentation | 89% | 55% |
| Video Tutorials | 52% | 79% |

**Willingness to Pay for Support:**

| Support Level | Monthly Willingness | % Interested |
|---------------|---------------------|--------------|
| Community (free) | €0 | 100% |
| Email Support | €50-100 | 45% |
| Priority Support | €200-500 | 28% |
| Dedicated Support | €1000+ | 12% |

**Implications:**
- Community building is strategic investment
- Tiered support model viable
- Free tier drives adoption, paid tiers monetize

---

### Finding 6: Performance Matters More Than Expected

**Evidence:**
- 67% have faced performance complaints from citizens
- 45% measure and monitor website performance
- 78% consider performance in theme selection

**Performance Priorities:**

| Metric | Awareness | Actively Measured |
|--------|-----------|-------------------|
| Page Load Time | 89% | 52% |
| Mobile Performance | 76% | 34% |
| Accessibility Score | 82% | 45% |
| SEO Score | 71% | 28% |
| Core Web Vitals | 45% | 18% |

**Implications:**
- Performance education opportunity
- Built-in performance monitoring valued
- Mobile performance increasingly important

---

### Finding 7: Decision-Making Process is Complex

**Evidence:**
- Average decision timeline: 3-6 months
- Average stakeholders involved: 5-8 people
- 67% require formal procurement process

**Decision Criteria Ranking:**

| Criterion | Importance (1-10) |
|-----------|-------------------|
| Compliance | 9.8 |
| Cost | 8.5 |
| Vendor Stability | 8.2 |
| Support Quality | 7.9 |
| Feature Completeness | 7.6 |
| Ease of Use | 7.4 |
| Community Size | 6.8 |
| Brand Recognition | 5.9 |

**Implications:**
- Sales cycle is long; nurture required
- Multiple stakeholders need different messaging
- Compliance is table stakes, not differentiator

## User Journey Maps

### Journey 1: Theme Evaluation

**Stage 1: Awareness**
- **Trigger:** New project requirement, compliance issue, migration need
- **Actions:** Google search, ask colleagues, browse GitHub
- **Thoughts:** "What are my options?" "What do others use?"
- **Emotions:** 😟 Overwhelmed by choices
- **Opportunity:** SEO, community presence, referrals

**Stage 2: Research**
- **Actions:** Read documentation, check GitHub, compare features
- **Thoughts:** "Is this compliant?" "Can my team use this?"
- **Emotions:** 😐 Cautiously optimistic
- **Opportunity:** Clear compliance claims, comparison guides

**Stage 3: Trial**
- **Actions:** Install theme, build test page, evaluate DX
- **Thoughts:** "Does this work as advertised?" "How's the DX?"
- **Emotions:** 😊 Hopeful, 😰 Anxious about issues
- **Opportunity:** Quick start guide, demo content

**Stage 4: Decision**
- **Actions:** Team discussion, stakeholder approval, procurement
- **Thoughts:** "Is this the right choice?" "What are the risks?"
- **Emotions:** 😰 Anxious, 😌 Relieved when decided
- **Opportunity:** Case studies, references, guarantees

**Stage 5: Onboarding**
- **Actions:** Team training, first project setup, customization
- **Thoughts:** "How do we use this effectively?"
- **Emotions:** 😓 Stressed, 🤓 Learning
- **Opportunity:** Training programs, templates

---

### Journey 2: Compliance Verification

**Stage 1: Planning**
- **Actions:** Review requirements, plan audit approach
- **Thoughts:** "What do we need to check?"
- **Emotions:** 😰 Overwhelmed
- **Opportunity:** Compliance checklists

**Stage 2: Testing**
- **Actions:** Run automated tools, manual testing
- **Thoughts:** "Will we pass?" "What will fail?"
- **Emotions:** 😰 Anxious
- **Opportunity:** Pre-audit tools

**Stage 3: Remediation**
- **Actions:** Fix issues, retest, document
- **Thoughts:** "This is taking forever"
- **Emotions:** 😓 Frustrated
- **Opportunity:** Pre-validated components

**Stage 4: Certification**
- **Actions:** Submit documentation, await approval
- **Thoughts:** "Will they approve?"
- **Emotions:** 😰 Nervous, 😌 Relieved
- **Opportunity:** Audit-ready documentation
